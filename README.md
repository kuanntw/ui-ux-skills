# Web UI/UX 設計規範

本規範整理自常見且已有明確定義的 Web UI/UX、RWD、可及性與設計系統原則，並加入本專案的資訊層級與註解文字規範。目標是讓每個畫面、每個區塊、每個元件在不同螢幕尺寸下都能清楚傳達任務重點，避免資訊互相干擾。

## 1. 外部已有定義的基準

以下規範與方法已被廣泛定義，可作為本文件的依據：

| 主題 | 已有定義 / 來源 | 納入本規範的重點 |
| --- | --- | --- |
| Web 可及性 | [W3C WCAG 2.2](https://www.w3.org/TR/WCAG22/) | 可感知、可操作、可理解、穩健；文字對比、非文字對比、色彩不可作為唯一提示、Reflow、標籤與指示、目標尺寸等。 |
| 視覺層級 | [Nielsen Norman Group - Visual Design Principles](https://media.nngroup.com/media/articles/attachments/Principles_Visual_Design-A4.pdf) | 用大小、對比、位置、間距、群組與視覺重量引導使用者先看最重要的資訊。 |
| RWD / 版面 | [Material Design - Responsive UI](https://m1.material.io/layout/responsive-ui.html)、[Material Design - Understanding Layout](https://m2.material.io/design/layout/understanding-layout.html) | 使用一致的 grid、spacing、breakpoints 與可重排的 layout；小尺寸畫面避免同時塞入多層資訊。 |
| 色彩系統 | [Material Design - Color System](https://m2.material.io/guidelines/style/color.html) | 色彩需有語意、品牌主副色、狀態色、可讀性與可互動提示。 |
| 表單輔助文字 / Microcopy | [Mass.gov UX microcopy](https://www.mass.gov/info-details/user-experience-ux-microcopy)、[FanDuel Design System - Text fields](https://fanduel.design/formation/components/text-fields/) | Label 說明欄位是什麼；helper text 說明限制、格式、可見性、後果或下一步，不要重複 label。 |

## 2. 核心原則

1. **以任務為中心**：畫面優先支援使用者當下最重要的任務，不用裝飾性內容搶走注意力。
2. **每個畫面都要有資訊層級**：所有畫面與區塊都必須區分主要資訊、次要資訊、輔助資訊。
3. **一致且可預期**：相同功能、狀態、元件、間距、顏色與文案在整個產品中應保持一致。
4. **RWD 優先**：設計需先確認小螢幕、觸控操作與內容重排方式，再擴展到桌機大畫面。
5. **可及性不可後補**：對比、鍵盤操作、焦點、文字縮放、目標尺寸與錯誤提示都要在設計階段納入。
6. **文字是介面的一部分**：標籤、註解、按鈕、錯誤訊息、空狀態與提示都需具備明確目的。

## 3. 資訊層級規範

### 3.1 必須分類

每一個畫面、區塊、卡片、表單或資料列都必須先定義：

| 層級 | 定義 | 常見呈現 | 規範 |
| --- | --- | --- | --- |
| 主要資訊 | 使用者完成任務或理解狀態最需要先看到的內容。 | 頁面標題、核心數字、主要 CTA、目前狀態、關鍵表單欄位。 | 必須最容易被掃描；不可被裝飾、廣告、次要操作或過多輔助文字干擾。 |
| 次要資訊 | 支援主要資訊理解或決策的補充內容。 | 摘要、日期、分類、狀態說明、次要操作、篩選條件。 | 可見但不得搶過主要資訊；通常使用較低視覺重量。 |
| 輔助資訊 | 只有在需要時才使用的背景、限制、說明、備註或延伸資訊。 | helper text、caption、tooltip、備註、詳細規則、次要 metadata。 | 應以較低強度呈現；必要時可收合、延後顯示或放在說明區。 |

### 3.2 設計檢查

每個畫面交付前必須回答：

- 使用者第一眼應該看到什麼？那是不是畫面上視覺重量最高的元素？
- 次要資訊是否真的支援主要任務，而不是干擾？
- 輔助資訊是否過多、過早出現，或和主要資訊競爭？
- CTA 是否分出主操作、次操作與危險操作？
- 如果移到手機寬度，主要資訊是否仍然優先出現？
- 如果使用者只掃描標題、按鈕與欄位 label，是否仍能理解任務？

### 3.3 視覺手段

可用以下方法建立層級，但需一致使用：

- **尺寸**：主要標題、核心數字與主要 CTA 可更大；輔助資訊不得比主要資訊更醒目。
- **字重**：主要資訊可用 semibold / bold；輔助資訊通常使用 regular。
- **顏色**：主要操作可使用主色；輔助資訊用中性色；危險操作只用於真正破壞性行為。
- **位置**：主要資訊放在視線起點或區塊頂部；次要資訊靠近相關主要資訊；輔助資訊放在附近但不阻斷流程。
- **留白**：使用間距分群，相關資訊靠近，不相關資訊拉開。
- **容器**：卡片、分隔線、背景色或邊框只用於建立群組，不應過度切割。

## 4. Web 版面與 RWD 規範

### 4.1 Breakpoints

建議以內容與常見裝置尺寸定義 breakpoints，預設可採：

| 名稱 | 寬度 | 設計重點 |
| --- | --- | --- |
| Mobile | `< 600px` | 單欄、優先呈現主要資訊、避免同時顯示 master-detail 兩層內容。 |
| Tablet | `600px - 959px` | 可視內容量增加，但仍需避免過多欄位與密集資訊。 |
| Desktop | `960px - 1279px` | 可使用 2 欄或主內容 + 側欄。 |
| Large desktop | `>= 1280px` | 可提高資訊密度，但正文寬度需限制，避免過長行寬。 |

### 4.2 RWD 行為

- 優先採用 fluid layout，再用 breakpoints 修正真正破版或資訊層級改變的情境。
- 手機畫面以單欄為預設；卡片、表單、列表應垂直排列。
- 桌機可使用多欄，但不可讓次要欄位搶走主要任務焦點。
- 內容重排後，閱讀順序、鍵盤焦點順序與 DOM 順序需維持合理。
- 重要操作在手機上需容易觸達；避免只把主要 CTA 放在桌機側欄。
- 不得要求使用者在一般內容中同時水平與垂直捲動；資料表等例外需提供清楚替代操作。
- 文字放大到 200% 時，不應造成內容或功能遺失。

### 4.3 間距與對齊

- 建議使用 4px 或 8px spacing scale，例如 4 / 8 / 12 / 16 / 24 / 32 / 48 / 64。
- 同一層級的元件使用一致間距；不同層級使用明顯間距差建立分群。
- 表單欄位與 label、helper text、error message 應有固定關係，不要在不同畫面任意變動。
- 主要內容區應有最大寬度，避免桌機長行閱讀困難。

## 5. 色彩規範

### 5.1 色彩角色

色彩必須有角色，不應只因「好看」而新增：

| 角色 | 用途 | 注意事項 |
| --- | --- | --- |
| Primary | 主要品牌色、主要 CTA、目前選取狀態。 | 一個畫面中不可過量使用，否則主次不明。 |
| Secondary | 次要品牌或次要操作輔助。 | 不應與 primary 競爭。 |
| Neutral | 背景、文字、邊框、分隔、容器。 | 需覆蓋多階層文字與表面。 |
| Success | 成功、完成、正向狀態。 | 不可只靠綠色傳達成功，需搭配文字或圖示。 |
| Warning | 需注意、可恢復風險。 | 文案需說明風險與建議行動。 |
| Danger / Error | 錯誤、刪除、不可逆或破壞性操作。 | 僅用於真正錯誤或危險情境，避免濫用。 |
| Info | 中性提示、系統資訊。 | 不應比主要任務醒目。 |

### 5.2 對比與可及性

- 一般文字與背景對比至少 **4.5:1**。
- 大型文字與背景對比至少 **3:1**。
- UI 元件邊界、焦點框、圖示、圖表中具意義的非文字元素，與相鄰顏色對比至少 **3:1**。
- 顏色不可作為唯一資訊提示；錯誤、成功、警告、連結、圖表類別都需搭配文字、圖示、底線、形狀或 pattern。
- Disabled 狀態可以降低對比，但仍需避免讓可互動與不可互動狀態混淆。
- 深色模式與淺色模式都需重新檢查對比，不可直接反轉色票。

### 5.3 使用限制

- 不要在同一畫面使用多個高飽和主色競爭注意力。
- 不要用紅色呈現一般提示，避免與錯誤或危險混淆。
- 不要讓背景裝飾色影響文字可讀性。
- 不要用低對比灰字承載必要資訊。
- 圖表不可只靠紅綠區分；需提供圖例、標籤、pattern 或形狀差異。

## 6. 不同應用領域的色系建議

> 重要原則：以下是依研究與常見產業語意整理出的「建議起點」，不是硬性規定。最後主色調仍應由使用者、品牌定位、目標客群、文化脈絡與可及性測試結果決定。若使用者已有品牌色，應優先保留品牌識別，再透過輔助色、狀態色、對比與版面層級補足 UX。

### 6.1 研究依據與使用限制

- 色彩會影響品牌人格、可信度、滿意度與購買意圖，但效果會受到產品類型、品牌定位、文化、情境、飽和度與明度影響；不能簡化成「某色一定代表某情緒」。可參考 Labrecque & Milne 的品牌人格色彩研究、Bottomley & Doyle 的產品類型與 logo 色彩適配研究，以及 Cyr、Head、Larios 的網站色彩、信任與滿意度研究。
- 行銷研究指出，色彩可影響注意力、情緒、等待感與食慾等感受；但跨文化研究也提醒，不同文化對同一顏色可能有不同解讀，因此跨國產品需做在地驗證。
- 實務上應把色彩視為「品牌語意 + 資訊層級 + 可及性 + 情境測試」的組合，而不是單一心理暗示。
- CTA 顏色不應只看「紅色或綠色哪個轉換率高」，而應看它在當前畫面是否有足夠對比、是否符合任務語意、是否不與錯誤/成功狀態混淆。

參考研究與資料：

- [Labrecque & Milne, *Exciting Red and Competent Blue: The Importance of Color in Marketing*](https://link.springer.com/article/10.1007/s11747-010-0245-y)
- [Bottomley & Doyle, *The interactive effects of colors and products on perceptions of brand logo appropriateness*](https://journals.sagepub.com/doi/abs/10.1177/1470593106061263)
- [Cyr, Head & Larios, *Colour appeal in website design within and across cultures*](https://experts.mcmaster.ca/scholarly-works/58866)
- [Singh, *Impact of Color on Marketing*](https://www.emerald.com/insight/content/doi/10.1108/00251740610673332/full/html)
- [Aslam, *Are You Selling the Right Colour? A Cross-cultural Review of Colour as a Marketing Cue*](https://www.tandfonline.com/doi/abs/10.1080/13527260500247827)
- [Sundar & Kellaris, *How Logo Colors Influence Shoppers’ Judgments of Retailer Ethicality*](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=2710328)
- [W3C WCAG 2.2](https://www.w3.org/TR/WCAG22/)

### 6.2 領域建議色系

| 應用領域 | 常見使用者期待 | 建議主色調 | 建議配色方向 | 適合原因 / 研究支持 | 注意事項 |
| --- | --- | --- | --- | --- | --- |
| 金融、銀行、保險、FinTech | 安全、可信、穩定、專業。 | 深藍、藍綠、深綠。 | 深藍 + 白 / 淺灰；藍綠 + 中性色；少量綠色用於收益或成功狀態。 | 藍色常被用於傳達能力、可信與專業；功能型產品也較適合藍、黑、綠、灰等 functional colors。 | 不要用紅色作為一般主色，容易和虧損、警示、錯誤混淆；若品牌使用紅色，需明確區分 danger/error 狀態。 |
| 醫療、健康、照護 | 安心、潔淨、可靠、低焦慮。 | 藍、青綠、綠、低飽和中性色。 | 藍 / 青綠 + 白；綠 + 米白；高對比文字。 | 藍綠系常被視為平靜、乾淨、健康與可信；醫療情境需要降低壓力並維持高度可讀性。 | 不要只靠紅綠呈現病況；警示、異常、危急需搭配文字、圖示與明確層級。 |
| 教育、學習、知識平台 | 清楚、可信、鼓勵、專注。 | 藍、青綠、紫、溫和橘。 | 藍 + 淺灰；青綠 + 白；紫作為創意或進階學習輔色；橘用於鼓勵與進度。 | 藍色有能力與可信感；紫色可傳達創意與想像；暖色可提高注意與動機。 | 避免整頁高飽和橘/黃造成疲勞；學習內容區應以可讀性優先。 |
| B2B SaaS、工具、生產力、管理後台 | 效率、秩序、穩定、可掃描。 | 藍、靛藍、藍灰、深灰。 | 藍 + neutral scale；狀態色清楚區分 success/warning/error/info；資料密集頁保持低干擾。 | 功能型與工具型產品適合 functional colors；藍色也常與 competent、reliable 的品牌人格連結。 | 不要讓品牌色污染所有資料視覺化；資料圖表應使用獨立且可辨識的 categorical palette。 |
| 電商、零售、促銷 | 吸引注意、行動、優惠、信任交易。 | 品牌主色可彈性；促銷常用紅、橘；信任與結帳可用藍/綠。 | 主品牌色 + 高對比 CTA；促銷色只用於優惠與限時；結帳流程使用較穩定色系。 | 暖色可提高喚起與注意；色彩會影響購買與品牌印象，但需和產品定位一致。 | 不要讓促銷紅與錯誤紅混淆；折扣、庫存、錯誤、價格需有不同語意。 |
| 食品、餐飲、外送 | 食慾、速度、親切、活力。 | 紅、橘、黃、綠。 | 紅/橘 + 暖中性色；綠用於健康、有機、蔬食；黃色作為活力輔色。 | 行銷研究常指出色彩可影響食慾、情緒與注意力；暖色通常較容易建立活力與刺激感。 | 黃色大面積使用可讀性差；健康餐飲若使用紅橘過重，可能和清爽/自然定位衝突。 |
| 永續、環保、農業、自然 | 自然、成長、責任、平衡。 | 綠、青綠、土色、米色。 | 綠 + 米白 / 棕；青綠 + 淺灰；少量黃色作陽光感。 | 綠色具有自然、成長、健康與環境聯想；研究也顯示綠色線索會影響 eco-friendly 感知。 | 避免過度使用綠色造成 greenwashing 感；永續主張需有內容證據支撐。 |
| 社群、通訊、社交產品 | 連結、信任、親和、活躍。 | 藍、紫、粉、橘。 | 藍 + 白建立信任；紫/粉增加個性與情感；橘用於互動提示。 | 藍色在網站偏好與信任研究中常有正向結果；感官/社交型產品可使用較具情緒與自我表達的色彩。 | 需注意不同文化與性別刻板印象；不要用粉色預設特定族群。 |
| 娛樂、遊戲、影音、活動 | 沉浸、刺激、情緒、差異化。 | 紫、紅、黑、霓虹藍/綠。 | 深色背景 + 高彩度 accent；紫/紅營造戲劇性；霓虹色只作重點。 | 紅與高飽和暖色可提高喚起；紫、黑與高對比色能建立感官與風格感。 | 高彩度與深色模式需檢查文字對比與視覺疲勞；不可犧牲可讀性。 |
| 美妝、時尚、精品、生活風格 | 審美、個性、質感、身份認同。 | 黑白、米色、金色、紫、粉、酒紅。 | 黑白 + 單一高級 accent；米色 + 金；紫/粉用於柔和或創意定位。 | 感官/社交型產品可使用較具 pleasure、identity、status 的色彩；近期研究也指出黑白 logo 可提升部分品牌的 status 感。 | 金色與低對比米色容易造成可讀性不足；精品感不可只靠低對比小字。 |
| 政府、公共服務、非營利 | 公信力、清楚、可近用、包容。 | 藍、深綠、深灰、白。 | 深藍 + 白；深綠 + neutral；高對比狀態色。 | 公共服務需以信任、清楚與可及性為優先；藍與灰常能降低情緒干擾並提供穩定感。 | 不要使用過度品牌化或低對比色；高齡者與輔具使用者需求需優先。 |
| 開發者工具、資安、技術平台 | 專業、精準、控制、可靠。 | 深藍、靛藍、深灰、黑、綠。 | 深色 / 淺色雙模式；藍或綠作主色；狀態色要和 log severity 區分。 | 功能型產品適合 functional colors；深色系可支援長時間工作情境，但仍需可讀性測試。 | 終端機綠不應同時代表 success 與一般資訊；warning/error 必須明確分離。 |
| 兒童、親子、創意工具 | 安全、友善、探索、活力。 | 黃、橘、青綠、粉、紫。 | 明亮但低刺激的多色 palette；主色 + 2-3 個輔色；大面積背景降低飽和度。 | 暖色與多色可營造活力與探索；紫/粉/青綠可支援創意與親和感。 | 避免過度鮮豔造成注意力分散；教育與兒童產品仍需清楚資訊層級。 |

### 6.3 選定主色調的流程

1. **先問使用者 / 品牌方**：是否已有品牌色、禁用色、文化禁忌或競品差異化需求。
2. **定義品牌人格**：例如可信、活潑、專業、創意、精品、照護、速度、穩定。
3. **選 1 個主色，不超過 2 個品牌輔色**：避免多個高飽和色互相競爭。
4. **建立 neutral scale**：文字、背景、邊框、分隔線與容器應主要依靠中性色建立可讀性。
5. **建立狀態色**：success、warning、error、info 不應被品牌色覆蓋。
6. **檢查對比與色盲情境**：至少符合本文件的 WCAG 對比要求。
7. **做情境測試**：在首頁、表單、資料表、CTA、錯誤狀態、空狀態、深色模式與手機畫面中檢查。
8. **由使用者確認最終主色調**：設計者可提出建議與風險，但最終色彩決策需回到使用者/品牌方。

## 7. 表單、註解與 Microcopy 規範

### 7.1 Label、Placeholder、Helper text 的責任

| 文字類型 | 應該做什麼 | 不應該做什麼 |
| --- | --- | --- |
| Label | 說明欄位是什麼。 | 不要塞入規則、範例或長說明。 |
| Placeholder | 可提供簡短格式範例，例如 `example@email.com`。 | 不要重複 label；不要承載必須知道的規則，因為輸入後會消失。 |
| Helper text / Caption / 註解 | 說明限制、格式、可見性、用途、後果或下一步。 | 不要寫空泛文字，也不要只重複欄位名稱。 |
| Error message | 說明哪裡錯、為什麼錯、如何修正。 | 不要只寫「格式錯誤」或「請重新輸入」。 |

### 7.2 有意義註解規範

註解必須回答使用者可能會有的具體問題，例如：限制是什麼、誰看得到、會發生什麼、是否可修改、何時生效。

#### 密碼欄位

- 不佳：`設定密碼`
- 較佳：`密碼需至少 12 碼，並包含英文、數字與符號。`
- 若有特殊限制，需明確寫出：`不可包含帳號、姓名或連續 3 個相同字元。`

#### 備註欄位

- 不佳：`備註`
- 較佳：`此備註僅自己可見，不會顯示給客戶。`
- 若是公開備註：`此備註會顯示給所有協作者，請勿填寫敏感資料。`

#### 日期或時間欄位

- 不佳：`選擇日期`
- 較佳：`系統會以你所在時區顯示，截止後將無法再修改。`

#### 上傳欄位

- 不佳：`上傳檔案`
- 較佳：`支援 PDF、JPG、PNG，單一檔案上限 10 MB。`

### 7.3 撰寫原則

- 先寫使用者要完成的事，再寫系統限制。
- 用使用者語言，不用內部代碼、資料表欄位名或工程術語。
- 指示需具體，避免「請填寫正確資料」這種無法行動的文字。
- 有風險或可見性問題時必須明說，例如「所有人可見」、「僅管理員可見」、「刪除後無法復原」。
- 需要格式時提供範例，例如 `YYYY/MM/DD`、`example@email.com`。
- 錯誤訊息應靠近錯誤欄位，並保留 label，避免使用者忘記欄位目的。

## 8. 元件互動規範

- 可點擊元素需有 hover、focus、active、disabled、loading、error 等必要狀態。
- 鍵盤焦點必須可見，且不可被 sticky header、modal 或浮層遮住。
- 觸控目標至少符合 WCAG 2.2 的 24 x 24 CSS px；若是主要操作，建議使用更大的觸控範圍。
- 主要 CTA 每個區塊通常只保留一個；多個同級 CTA 會削弱決策。
- 危險操作需明確命名，例如「刪除專案」優於「確定」。
- Loading 狀態需說明正在處理什麼；長時間處理需提供進度、取消或稍後通知方式。

## 9. 檢核清單

設計、審稿或開發交付前，至少檢查以下項目：

### 9.1 資訊層級

- [ ] 每個畫面已標註主要資訊、次要資訊、輔助資訊。
- [ ] 主要資訊沒有被次要操作、裝飾或過多說明干擾。
- [ ] 每個區塊都有清楚標題或可理解的上下文。
- [ ] 每個 CTA 已區分主要、次要、危險或取消。

### 9.2 RWD

- [ ] Mobile、Tablet、Desktop 都有合理版面與閱讀順序。
- [ ] 手機不需要在一般內容中水平捲動。
- [ ] 主要資訊在小螢幕仍優先出現。
- [ ] 文字放大與內容重排不會遺失功能。

### 9.3 色彩

- [ ] 一般文字對比至少 4.5:1，大型文字至少 3:1。
- [ ] 非文字但有意義的 UI 元素對比至少 3:1。
- [ ] 沒有任何狀態只靠顏色傳達。
- [ ] 狀態色使用一致且未被濫用。

### 9.4 表單與文字

- [ ] Label 說明欄位是什麼。
- [ ] Placeholder 不重複 label，且不承載必要資訊。
- [ ] Helper text / caption / 註解說明限制、格式、可見性、用途或後果。
- [ ] Error message 說明錯誤原因與修正方式。
- [ ] 涉及公開、共享、刪除、付款、權限與敏感資料時，已明確說明影響範圍。

## 10. 設計交付要求

每個畫面或元件交付時，需包含：

1. **資訊層級標註**：主要資訊、次要資訊、輔助資訊。
2. **RWD 行為**：至少描述 mobile / tablet / desktop 的排列與內容優先順序。
3. **色彩角色**：使用的 primary、secondary、neutral、status colors 與對比檢查結果。
4. **文字規格**：label、placeholder、helper text、error message、empty state、button text。
5. **互動狀態**：default、hover、focus、active、disabled、loading、error、success。
6. **可及性備註**：鍵盤操作、焦點順序、ARIA / semantic HTML 需求、圖片替代文字。
